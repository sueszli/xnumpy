title: profile-guided kernel hot-swap via jit-compiled user-schedulable programs


1. motivation

- ml kernels run on diverse and changing hardware, data shapes, and workload phases. schedules chosen offline can't adapt.
- pytorch/jax: jit but schedules are compiler-internal. no user control, no correctness guarantees on transformations.
- halide/tvm/exo: user-facing, correct-by-construction schedule transformations. but they emit c. recompilation takes seconds, adaptation at runtime is impractical.
- gap: no system combines correct schedule transformations with fast enough recompilation to adapt during a live workload.
- exo separates algorithm from schedule. each transformation is semantics-preserving by construction.
- exojit compiles exo to native code via llvm in-process in milliseconds. no gcc, no leaving python.
- together: profile -> diagnose bottleneck -> select transformation -> recompile -> hot-swap -> repeat. all at runtime.


2. what exojit exposes at runtime

data:
- tensor shapes (resolved at call time), shape change frequency
- tensor sparsity (density ratio, structure)
- value distributions, pointer alignment

hardware:
- cache sizes, simd width, core count/topology, cpu features
- thermal state, co-tenancy pressure

execution:
- wall-clock time per kernel invocation
- perf counters: l1/l2/l3 cache misses, ipc, branch mispredicts, tlb misses
- simd utilization, memory bandwidth
- compilation time (amortization budget)

workload (training-specific):
- convergence rate, gradient magnitudes
- training phase, per-layer bottleneck distribution


3. research directions

systems:
- profile-guided transformation selection: map perf counters to exo schedule transformations (high l1 misses -> smaller tiles, low ipc -> unroll, low simd util -> register staging, high tlb misses -> loop reorder). evaluate: profile-guided vs blind search, candidates needed to find good schedule.
- hot-swap during live training/inference: background thread searches, swaps function pointer on improvement, zero downtime. differential test old vs new on same input. evaluate: swap latency and overhead.
- shape specialization: bake shapes as constants, llvm unrolls/constant-folds/vectorizes. cache compiled variants by shape signature. recompile on shape change. evaluate: specialized vs generic kernel speedup.
- hardware-adaptive dispatch: detect machine at startup, select schedule. one package runs optimally everywhere. evaluate: cross-machine portability vs hand-tuned per-target.
- cross-kernel co-optimization: jointly optimize schedule chains (matmul -> rmsnorm) since one schedule affects memory layout seen by the next. evaluate: joint vs independent optimization.
- memory allocation strategy switching: stack vs heap vs pool based on observed allocation patterns and buffer reuse.
- sparsity-aware schedules: recompile with sparse iteration when density drops. requires extending exo with sparse primitives.
- precision adaptation: recompile same algorithm at f64/f32/f16 per-kernel based on runtime accuracy requirements. evaluate: throughput vs numerical accuracy tradeoff.

ml:
- learned cost model: train online from collected (schedule, shape, counters, runtime) tuples. predict runtime without benchmarking. evaluate: prediction accuracy vs sample count.
- cost model transfer: schedule representation shared across kernel types — model trained on matmul may generalize to attention. evaluate: transfer accuracy.
- search via bayesian optimization / rl over the schedule space. evaluate: sample efficiency vs grid/random search.
- llm-guided schedule exploration: llm proposes schedules, exo type system validates correctness, jit benchmarks. evaluate: llm suggestions vs random vs expert.
- workload phase detection: invest autotuning budget during convergence plateaus, back off during fast convergence. evaluate: phase detection accuracy, training time saved.
- gradient-aware precision: shrinking gradients -> lower precision kernels automatically. evaluate: convergence impact.
- learned bottleneck diagnosis: classifier maps counter vectors to transformations, replacing hand-written rules. evaluate: classification accuracy, end-to-end speedup vs rule-based.

both:
- the collected (schedule x shape x hardware -> runtime) dataset as a standalone contribution for offline research.
- end-to-end adaptive system: combine phase detection, cost model, profile-guided search, and hot-swap. evaluate: micro-gpt training/inference with continuous adaptation vs frozen kernels (pytorch, jax, numba, static exo).
